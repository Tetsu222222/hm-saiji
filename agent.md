## 使い方

- 質問には日本語で丁寧に回答します。
- 変更は既存のプロジェクト構造と設定を壊さないように行います。
- 新規ファイルや機能追加が必要な場合は、まずユーザーに確認します。

## 注意点

- `frontend/` と `backend/` はそれぞれ独立したサブプロジェクトです。
- 既存の `package.json` や `tsconfig.json` の設定を尊重します。
- 大きな設計変更や不明点がある場合は、必ずユーザーに確認します。

## コミュニケーション

- 簡潔かつ事実に基づいて回答してください。
- 絵文字はあまり使用しないでください。
- 過度な賞賛や褒め言葉は不要です。動作確認結果と客観的な情報のみを報告してください。
- コードやファイルの作成・更新を行う際は、まず相談ベースで提案し、必ず許可（承認）を得てから実行してください。

## デザイン方針

- 装飾的なデザイン（過度なグラデーションやアニメーションなど）は避けてください。
- シンプルで機能的、かつ使いやすさを最優先としたUI設計（Vanilla CSS）にしてください。
- レイアウトは視認性と実用性を重視してください。

## 開発・運用ルール

- 2台のPCで同期開発を行うため、ポータブルなコード構成を維持してください。
- デプロイはGitHubへのプッシュ（GitHub Actions）による自動化を基本としてください。

## 開発フロー（2台のPC同期ルール）

2台のPCで交互に作業を行うため、同期ミスやコードの衝突を防ぐ以下の「黄金ルーティン」を徹底します。AIがコードを提案・変更する際も、この流れに沿ってサポートします。

### 1. 作業開始時のルーチン（必ず行う）
- 最新コードの取得（Pull）:
  作業を開始するPCでエディタを開いたら、まず最初に必ず `git pull`（または「同期」ボタンのクリック）を行い、もう1台のPCで行った最新の変更を取り込みます。
  ※これを行わずにコードを書き始めると、衝突（コンフリクト）の原因になります。

### 2. 作業中のルール
- 作業用ブランチの作成:
  大きな変更や新機能を作る際は、本番用ブランチ（`main` や `master`）で直接作業せず、作業用ブランチ（例: `feature/xxx`）を作って作業します。
- こまめな保存（Commit）:
  キリの良いところでこまめにコミット（保存）します。

### 3. 作業終了時・PC切り替え時のルーチン（必ず行う）
- クラウドへの送信（Push）:
  その日の作業が終わる時、またはもう1台のPCに切り替える時は、必ずすべての変更をコミットし、`git push`（または「変更を送信」）を行ってGitHub（クラウド）にアップロードします。
- WIP（仕掛かり中）コミットの活用:
  「まだ動かないけれど、今日の作業はここまでにして、明日はもう1台のPCで続きをやりたい」という場合は、メッセージに `wip: 〇〇の実装途中` と書いてPushして構いません。

### 4. 衝突（コンフリクト）が発生した場合の対処
- 万が一、両方のPCで別々の変更をしてしまい衝突が発生した場合は、無理に上書きせず、現在の状態を整理してユーザーに解決方針を確認します。AIがサポートします。
